1 Contexto

O Setor de Análise da MZ NET é responsável pelo fluxo completo de entrada de novos clientes: da criação do lead até a abertura da O.S. de instalação no MK ERP. Atualmente todo esse processo é feito manualmente, passo a passo, pelo analista dentro do sistema.

Estamos mapeando quais etapas do fluxo podem ser automatizadas via API para integração com nosso sistema interno (CRM mznet.digital), eliminando o trabalho repetitivo e reduzindo erros de preenchimento. O objetivo deste documento é identificar:

  • Quais APIs já estão disponíveis no nosso pacote atual;
  • Quais precisam ser contratadas separadamente;
  • Quais endpoints não existem ou precisam de esclarecimento.

2 Fluxo do processo — Análise de ficha

O processo mapeado a partir do procedimento interno do setor:

1
Verificar cadastro do cliente por CPF/CNPJ
API: WSMKConsultaDoc.rule  ✓ Disponível
Se inativo: necessário reativar e atualizar endereço antes de continuar.
2
Criar cadastro (se não existir) ou atualizar dados
Criar: WSMKNovaPessoa.rule  ★ API especial
Atualizar: PUT /pessoas  ★ API Node.js
Inclui reativação de inativo, atualização de endereço e demais dados cadastrais.
3
Abrir Lead no CRM
WSMKNovaLead.rule  ★ API especial
Alternativa (lead + pré-cadastro em um passo): WSMKInserirLead.rule  ★ API especial
Vincula o produto solicitado, marca negócio fechado, define vendedor e vencimento.
4
Verificar viabilidade de endereço (portas disponíveis)
WSMKVerificaDispPorta.rule (lat/lon)  ★ API especial
Retorna caixas próximas com portas disponíveis. Reserva de porta: WSMKReservarPorta.rule.
5
Listar planos disponíveis
WSMKListaPlanoAcesso.rule / WSMKConsultaPlanos.rule  ✓ Disponível
6
Criar contrato (com plano, vencimento, forma de pagamento)
WSMKNovoContrato.rule  ★ API especial
Parâmetros necessários: CodigoPlanoAcesso, CodigoRegraVencimento, CodigoRegraBloqueio, CodigoFormaPagamento, CodigoProfilePagamento, CodigoMetodoFaturamento, CodigoPlanoContas.
7
Criar conexão (ponto de acesso, servidor concentrador)
WSMKCriarConexao.rule  ★ API especial
Listar pontos de acesso: WSMKConexaoPontoAcesso.rule  ✓ Disponível
Listar servidores (Hawuey Patricia etc.): WSMKConexaoServidorConcentrador.rule  ✓ Disponível
8
Criar O.S. de instalação
WSMKCriarOrdemServico.rule  ★ API especial
Tipo OS: Instalação e Ativação (ou + Wi-Fi Extend).
O texto da O.S. é gerado pelo Gerador de O.S. atual e passado em DescricaoProblema.
Requer: CodigoTipoOS, CodigoTecnico, CodigoGrupoServico — códigos não levantados ainda.
9
Abrir protocolo / atendimento
WSMKNovoAtendimento.rule  ✓ Disponível e funcionando em produção
Registro do operador via op_abertura já validado (atend. #268168).
10
Inserir comentário no atendimento (padrão de aprovação, negação, etc.)
WSMKAtendimentoComentario.rule  ✗ Bug ativo — HTTP 500
Aguardando resolução MK Solutions (chamado aberto — analista Jardel Henrique). Ver Seção 4.
11
Encaminhar atendimento para subprocesso (aprovado instalação gratuita / compra de equip.)
? Não encontrado na documentação disponível
Precisamos confirmar se existe endpoint para encaminhar um atendimento entre subprocessos via API.
12
Ativar contrato pós-instalação
WSMKAtivarContrato.rule  ★ API especial

★ O que são as APIs especiais?
Com base na documentação MK Solutions, os endpoints listados acima fazem parte de um pacote de webservices aprimorados, comercializados separadamente do pacote base. Gostaríamos de confirmar quais desses módulos precisam ser contratados, seus nomes comerciais e como proceder para ativação.

4 Perguntas para o MK Solutions

  • Q1 WSMKAtendimentoComentario.rule — módulo contratado?
    Em chamado anterior (analista Jardel Henrique), foi informado que "a API não está na nossa base". Precisamos confirmar: esse endpoint requer a contratação de um módulo adicional? Se sim, qual o nome comercial e como procedemos?
  • Q2 WSMKCriarOrdemServico.rule — módulo contratado?
    Da mesma forma, gostaríamos de confirmar se esse endpoint está incluído no pacote atual ou requer contratação separada.
  • Q3 APIs especiais — quais precisam ser contratadas?
    Dos endpoints listados na Seção 3.2 (WSMKNovaPessoa, WSMKNovaLead, WSMKNovoContrato, WSMKCriarConexao, WSMKAtivarContrato, WSMKVerificaDispPorta, WSMKReservarPorta), quais fazem parte do pacote base e quais requerem contratação adicional? Podemos receber uma lista com os módulos disponíveis e seus respectivos valores?
  • Q4 Encaminhamento de atendimento entre subprocessos via API?
    No nosso fluxo, após a análise da ficha o analista encaminha o atendimento para um subprocesso específico ("Aprovado Instalação Gratuita", "Aprovado Compra de Equipamento", "Negado", etc.). Existe algum endpoint que permita realizar esse encaminhamento via API?
  • Q5 Reativação de cliente inativo via API?
    Quando um cliente já possui cadastro mas está marcado como inativo, o processo atual exige reativação manual em Pessoas/Empresas. O endpoint PUT /pessoas cobre esse caso, ou existe um endpoint específico para reativação?
  • Q6 WSMKCriarOrdemServico.rule — CodigoTecnico obrigatório?
    Como a O.S. é aberta remotamente antes de escalonar para campo, não temos um técnico definido no momento da criação. É possível criar a O.S. sem CodigoTecnico e designar depois pelo painel ERP? Ou existe um cadastro de "técnico genérico / a designar" que pode ser usado como placeholder?
