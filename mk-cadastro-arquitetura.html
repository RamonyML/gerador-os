1 A questão central

O sistema do Felipe já possui o Kanban de fichas, o fluxo de análise e o trigger de finalização. O que precisa ser definido é como o sistema dele se comunica com o MK a partir desse trigger — quem faz as chamadas, onde ficam as credenciais e como o gerador de O.S. se encaixa no fluxo.

Existem dois caminhos:

Opção A ✓ Recomendada
Sistema do Felipe chama nossa Cloud Function
Felipe's system
└── trigger disparado
└── POST → nossa CF (mkCadastro)
└── payload: { status, ficha }
└── CF orquestra todas as chamadas MK
└── retorna resultado consolidado
  • ✓ Credenciais MK ficam no Secret Manager — Felipe não precisa lidar com isso
  • ✓ Toda a lógica de orquestração centralizada em um lugar
  • ✓ Gerador de O.S. roda na CF antes de criar a O.S. no MK
  • ✓ Atualiza sem mexer no sistema do Felipe
  • – Sistema do Felipe depende da nossa infra para as chamadas MK
Opção B ✗ Descartada
Sistema do Felipe chama o MK diretamente
Felipe's system
└── trigger disparado
└── chama MK diretamente
└── precisa das credenciais
  • ✗ Exige repassar credenciais MK para o sistema do Felipe
  • ✗ Duplica toda a lógica de integração que já existe
  • ✗ Qualquer mudança na API MK exige atualização nos dois sistemas

2 O contrato da API

O sistema do Felipe envia um único payload para nossa Cloud Function mkCadastro. A CF cuida de todo o resto.

Entrada — o que o sistema do Felipe manda

{
  "action":    "processar_ficha",
  "status":    "aprovada" // ou "negada" | "cancelada"
  "operador":  "uid do usuário no Firebase",
  "ficha": {
    "cpf":              "000.000.000-00",
    "nome":             "Nome do cliente",
    "bairro":           "Nome do bairro",      // determina Zona Leste/Oeste
    "plano": { "codigo": "...", "nome": "..." },
    "dataInstalacao":   "2026-07-01",
    "formaPagamento":   "...",
    "vencimento":       10,
    "temWifiExtend":    false,
    "tipoAprovacao":    "gratuita",            // ou "compra_equipamento" | "novos_dados"
    "motivo":           "preenchido só quando negada/cancelada"
  }
}

Saída — o que a CF retorna

{
  "success": true,
  "etapas": {
    "cadastro":    { "ok": true,  "codigoPessoa": 28903 },
    "lead":        { "ok": true,  "codigoLead":   "..."  },
    "contrato":    { "ok": true,  "codigoContrato": "..." },
    "conexao":     { "ok": true,  "codigoConexao": 35525 },
    "os":          { "ok": true,  "textoGerado":  "...", "codigoOS": "..." },
    "atendimento": { "ok": true,  "protocolo":    "268262"             },
    "comentario":  { "ok": false, "erro":          "HTTP 500 — aguardando MK"   },
    "subprocesso": { "ok": false, "erro":          "endpoint não confirmado"    }
  }
}
⚠ Resiliência por etapa
A CF executa as etapas em sequência e registra o resultado de cada uma individualmente. Se uma etapa falhar, as anteriores já confirmadas ficam registradas — o analista sabe exatamente onde o fluxo parou e o que foi criado no MK.

3 O gerador de O.S. de cadastro

O algoritmo do gerador roda dentro da Cloud Function, antes de chamar WSMKCriarOrdemServico. Felipe não precisa implementar nada no sistema dele — manda os dados da ficha, a CF gera o texto e envia direto para o MK.

✓ Fluxo do gerador
Dados da ficha → CF recebe → algoritmo gera texto da O.S. → texto vai em DescricaoProblema na chamada WSMKCriarOrdemServico → resultado volta no campo etapas.os.textoGerado da resposta.

4 Os três fluxos internos

Ao receber o payload, a CF identifica o status e executa o fluxo correspondente:

✓ Aprovada
WSMKConsultaDoc (verificar cliente) WSMKNovaPessoa (criar/reativar) WSMKNovaLead (abrir lead) WSMKNovoContrato (criar contrato) WSMKCriarConexao (criar conexão) WSMKCriarOrdemServico (criar O.S.) WSMKNovoAtendimento (protocolo) WSMKAtendimentoComentario (padrão) encaminhar subprocesso
9 chamadas em sequência
✗ Negada
WSMKAtendimentoComentario (motivo)
Fecha o protocolo existente com o texto do motivo informado pelo analista.
1 chamada
✗ Cancelada
WSMKAtendimentoComentario (motivo)
Fecha o protocolo existente com o texto do motivo informado pelo analista.
1 chamada

6 O que pode ser construído agora

Quase todo o fluxo de Aprovada depende de APIs especiais aguardando confirmação do MK. Mas a estrutura pode ser construída por completo — quando as APIs forem liberadas, conectam no orquestrador já pronto.
O quê Dependência MK?
Cloud Function mkCadastro com o orquestrador dos três fluxos ✓ Não — estrutura com mocks nas etapas bloqueadas

Etapa de autenticação + busca de cliente por CPF ✓ Não — APIs disponíveis
