1 Contexto

A MZ NET está desenvolvendo um sistema interno chamado Gerador de O.S., construído sobre Firebase (Cloud Functions + React). O objetivo é automatizar a abertura de protocolos, comentários e ordens de serviço no MK ERP a partir de formulários preenchidos pela equipe de suporte técnico, eliminando o processo de cópia manual.

A integração é feita via Cloud Functions (proxy seguro — o frontend nunca acessa a API MK diretamente). Autenticação via WSAutenticacao.rule com token de perfil webservice.

3 Bloqueio crítico — WSMKAtendimentoComentario.rule

3.1 O problema

Ao chamar o endpoint com os campos obrigatórios documentados (token, cd_atendimento, comentario), o servidor retorna HTTP 500:

GET /mk/WSMKAtendimentoComentario.rule ?sys=MK0 &token=<token_valido> &cd_atendimento=268137 &comentario=CLIENTE SEM BLOQUEIO... &tipo=2 → HTTP 500: "O objeto (Tabela) deve ter um valor definido!" javax.servlet.ServletException wfr.web.ExternalRulesServlet.process(SourceFile:321)

O atendimento 268137 foi criado com sucesso pelo próprio sistema segundos antes. O erro ocorre independentemente do parâmetro tipo (testado com e sem ele, com valores 1 e 2).

3.2 Nossa análise técnica

O erro "O objeto (Tabela) deve ter um valor definido!" é uma exceção do runtime WFR indicando que um objeto de banco de dados retornou nulo onde não deveria. Nossa hipótese: a regra tenta associar o comentário a um operador ERP a partir da sessão do webservice — e como tokens de webservice (WSAutenticacao.rule) não têm um usuário ERP diretamente vinculado na mesma tabela de sessões de operadores, a busca retorna nulo.

⚠ Questão arquitetural importante
No MK ERP, quando um comentário é inserido manualmente, ele sempre aparece com o nome do operador que o registrou. Nosso sistema precisa manter esse rastreamento — cada operador da MZ NET que abre um atendimento deve aparecer como autor dos comentários, não um usuário genérico do webservice.

3.3 O que precisamos do MK Solutions

  • Q1 Causa raiz do 500: O parâmetro user é, na prática, obrigatório mesmo sendo documentado como opcional? Ou há algum pré-requisito de configuração no perfil webservice que não estamos atendendo?
  • Q2 Mapeamento de operadores: Cada solicitação no nosso sistema é feita por um operador autenticado (Firebase Auth). Para associar o comentário ao operador correto no MK, precisamos mapear o usuário Firebase ao login ERP. Existe um endpoint de consulta de usuários MK por nome ou e-mail (ex: WSMKConsultaUsuario.rule)? Ou devemos manter uma tabela de mapeamento manual?
  • Q3 Alternativa de autenticação: Seria mais adequado autenticar via WSMKLoginCompleto.rule (com login + senha de um usuário ERP dedicado) em vez de WSAutenticacao.rule (token de webservice), especialmente para que os comentários tenham um operador associado?

5 Fluxo atual funcionando (para registro)

O fluxo de criação de protocolo com seleção de conexão está 100% operacional. Testado em ambiente real com protocolo 2606.25251 criado com sucesso.
1. Auth → token de sessão GET /mk/WSAutenticacao.rule?sys=MK0&token=<token_webservice>&password=<senha>&cd_servico=9999 2. Buscar cliente por CPF GET /mk/WSMKConsultaDoc.rule?sys=MK0&token=<sessao>&doc=<cpf_sem_formatacao> → Retorna: CodigoPessoa, Nome 3. Listar conexões do cliente GET /mk/WSMKConexoesPorCliente.rule?sys=MK0&token=<sessao>&cd_cliente=<CodigoPessoa> → Campo retornado: "codconexao" (lowercase) — não "CodigoConexao" 4. Abrir atendimento (com contrato + conexão) GET /mk/WSMKNovoAtendimento.rule?sys=MK0&token=<sessao> &cd_cliente=<cod>&cd_processo=<cod>&cd_classificacao_ate=<cod> &origem_contato=9&info=<texto> &cd_contrato=<cod>&conexao_associada=<codconexao> → Retorna: CodigoAtendimento, Protocolo ✅
